# CAS Organizational Workspace

مجموعه افزونه‌های اختصاصی **چدن‌آرا شمال** برای ایجاد یک فضای کار سازمانی یکپارچه، فارسی، راست‌چین و نقش‌محور روی **Odoo 19 Community**.

این مخزن فقط مجموعه‌ای از افزونه‌های جداگانه نیست؛ هسته‌های عمومی فرم، گردش‌کار، تأیید، اقدام، سند و تاریخ جلالی را در کنار ماژول‌های عملیاتی مکاتبات، دبیرخانه، شیفت، حضور، کاردکس و گزارش کار قرار می‌دهد و آن‌ها را از طریق Workspace اختصاصی CAS در یک تجربه کاربری واحد ارائه می‌کند.

> اصل بنیادین پروژه: **هیچ تغییری در هسته Odoo انجام نمی‌شود.** تمام قابلیت‌ها با Custom Addon، inheritance، APIهای استاندارد Odoo، OWL، QWeb، ACL و Record Rule پیاده‌سازی می‌شوند.

## مشخصات فنی

- سکوی هدف: `Odoo 19.0 Community`
- زبان رابط: فارسی
- جهت رابط: RTL
- معماری: ماژولار، نسخه‌پذیر، ممیزی‌پذیر و چندشرکتی
- رابط کاربری: OWL / QWeb / SCSS
- تاریخ: ذخیره استاندارد Odoo و نمایش/ورود جلالی
- استقرار: Custom Addons مستقل، بدون تغییر Odoo Core

## اهداف اصلی

- فرم‌های سازمانی پویا و نسخه‌دار
- فرم‌ساز دیداری Drag & Drop
- گردش‌کارهای گرافیکی و نسخه‌پذیر
- تأییدهای چندمرحله‌ای و تفویض اختیار
- کارتابل یکپارچه وظایف و SLA
- مکاتبات رسمی، ارجاع و رسید مشاهده
- دبیرخانه، وارده/صادره، PDF رسمی و امضا
- مدیریت اسناد، نسخه فایل، پیوند رکورد و OCR
- برنامه‌ریزی شیفت و الگوهای چرخشی
- ثبت حضور از دستگاه یا نگهبانی
- ورود مرحله‌ای Excel و رفع مغایرت‌ها
- کاردکس، مرخصی، مأموریت، اضافه‌کاری و قفل دوره
- گزارش کار روزانه و شیفت
- تاریخ جلالی در فرم، جستجو، Chatter، HR و گزارش‌ها
- Workspace اختصاصی، واکنش‌گرا و نقش‌محور

## اصول معماری

- Workspace مالک داده عملیاتی نیست و مدل‌های منبع را مصرف می‌کند.
- Action Hub فقط اطلاعات اقدام و لینک رکورد منبع را نگه می‌دارد.
- Form Builder و Workflow Designer فقط نسخه پیشنویس هسته مربوط را ویرایش می‌کنند.
- Approval مالک تصمیم رسمی است؛ Workflow مالک وضعیت و مسیر فرایند باقی می‌ماند.
- Document Core نسخه فایل و پیوند رکورد را مدیریت می‌کند؛ معنای کسب‌وکاری سند در ماژول منبع باقی می‌ماند.
- تاریخ جلالی فقط لایه ورود و نمایش است و مقدار ذخیره‌شده Odoo را تغییر نمی‌دهد.

## امنیت و ممیزی

- پنهان‌کردن منو یا دکمه به‌تنهایی امنیت محسوب نمی‌شود.
- لینک مستقیم، RPC، Import، Export و Download باید ACL و Record Rule را رعایت کنند.
- نسخه منتشرشده فرم و گردش‌کار قفل می‌شود.
- هر Submission و Workflow Instance به نسخه مشخص متصل می‌ماند.
- تاریخچه گردش‌کار، تأیید، اسناد، مکاتبات و حضور باید Append-only باشد.
- ابطال، جایگزینی و بازگشایی رسمی جای حذف یا بازنویسی سابقه را می‌گیرد.
- هر تصمیم مهم باید Actor، Timestamp و Reason داشته باشد.
- کنترل چندشرکتی باید در مدل، Workspace، Cron، Export و Download رعایت شود.

## نصب و ارتقا

نصب یک ماژول:

```bash
./odoo-bin -d <database> -i <module_name> --stop-after-init
```

ارتقای یک ماژول:

```bash
./odoo-bin -d <database> -u <module_name> --stop-after-init
```

اجرای آزمون‌های یک ماژول:

```bash
./odoo-bin -d <database> --test-enable --test-tags /<module_name> -u <module_name> --stop-after-init
```

ترتیب پیشنهادی نصب:

1. `cas_core`
2. خانواده `cas_jalali_*`
3. هسته فرم و Runtime
4. هسته گردش‌کار، Designer، Approval و Action Hub
5. اسناد و مکاتبات
6. شیفت، حضور، کاردکس و گزارش کار
7. `cas_workspace`

## قواعد توسعه

1. هسته Odoo را تغییر ندهید.
2. هر قابلیت جدید را در یک ماژول با مرز مسئولیت روشن پیاده‌سازی کنید.
3. نام مدل عمومی، XML ID، وضعیت و Transition را بدون Migration تغییر ندهید.
4. تغییر Schema یا Lifecycle باید همراه آزمون Regression و اصلاح مستندات باشد.
5. کنترل سمت مرورگر جای اعتبارسنجی سمت سرور را نمی‌گیرد.
6. از `sudo()` برای دورزدن مشکلات دسترسی استفاده نکنید.
7. وابستگی‌های ثالث را پیش از اضافه‌کردن بررسی کنید.
8. رمز عبور، Token، کلید خصوصی، Dump پایگاه داده یا Filestore را Commit نکنید.

## مستندات

هر ماژول می‌تواند شامل این منابع باشد:

- `README.md`: معرفی و راهنمای سریع
- `docs/`: معماری، راهنمای استفاده و مرجع فنی
- `tests/`: آزمون‌های خودکار و سناریوهای Regression

برای درک کامل یک قابلیت، مستند معماری ماژول، مدل‌ها، Security، Viewها، Cronها و آزمون‌های همان ماژول را با هم بررسی کنید.
