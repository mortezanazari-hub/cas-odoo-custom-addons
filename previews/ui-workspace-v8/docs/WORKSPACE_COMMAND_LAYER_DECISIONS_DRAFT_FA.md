# پیش‌نویس تصمیمات طراحی Workspace و CAS Command Layer

> وضعیت: پیش‌نویس برای ارزیابی در Prototype v5. پس از تأیید تجربه کاربری، نسخه نهایی در مستندات محصول GitHub ثبت می‌شود.

## جهت محصول
- CAS یک پوسته تزئینی برای Odoo نیست؛ لایه تجربه کاری یکپارچه و اقدام‌محور است.
- Odoo موتور داده، امنیت، مدل‌ها و منطق Backend باقی می‌ماند.
- رابط CAS باید پیچیدگی ساختار ماژول‌های زیرین را از دید کاربر پنهان کند.

## Workspace
- Workspace مرکز فرمان شخصی و نقطه شروع روز کاری است، نه Dashboard آماری.
- ترتیب محتوا: اقدام‌های امروز، وضعیت، سپس تحلیل و KPI.
- Briefing متنی و عملیاتی جایگزین کارت‌های عددی بی‌معنا می‌شود.
- چیدمان و محتوا بر اساس نقش کاربر تغییر می‌کند.

## تفکیک مفاهیم
- «کارهای من»: آنچه باید انجام شود.
- «تقویم»: زمان انجام کار یا رویداد.
- «گزارش فعالیت»: آنچه واقعاً انجام شده است.
- این سه مفهوم مستقل ولی قابل ارتباط و تبدیل هستند.

## کارهای من
- استفاده نکردن از To-do و Project استاندارد Odoo.
- رابط فهرست‌محور، ساده و بدون Kanban.
- افزودن سریع با حداقل فیلد اجباری.
- امکان تبدیل اختیاری کار به رویداد تقویم یا گزارش فعالیت.

## تقویم
- صفحه مستقل روز/هفته/ماه و ویجت فشرده در Workspace.
- بررسی استفاده از زیرساخت مفید Calendar بدون استفاده از ظاهر پیش‌فرض.
- عدم تغییر هسته Odoo.

## Command Layer
- شامل Workspace، Briefing، Action Center، جست‌وجوی سراسری، کارهای من، تقویم، اعلان‌ها و تاریخچه اخیر.
- منطق تخصصی واحدها داخل Command Layer قرار نمی‌گیرد.
- هر ماژول تخصصی Action و Widget خود را معرفی می‌کند و Command Layer فقط آن را کشف و نمایش می‌دهد.
