# تصمیم‌های محصول Workspace در نسخه ۷

## خط مبنا

این تصمیم‌ها بر پایه `CAS_UI_Prototype_V4` ثبت شده‌اند. نسخه‌های ۵ و ۶ Release رسمی مستقل محسوب نمی‌شوند و صرفاً Iterationهای داخلی طراحی بوده‌اند.

## تصمیم‌های قطعی

### ۱. Workspace جایگزین Odoo Core نیست

تمام پیاده‌سازی باید در Custom Addons انجام شود و هیچ تغییری در هسته Odoo مجاز نیست.

### ۲. Workspace مالک داده کسب‌وکاری نیست

Workspace فقط مسئول موارد زیر است:

- Shell
- Navigation
- Dashboard
- Widget
- Drawer
- Route Resolution
- Action Dispatch

مالکیت داده و منطق تخصصی در ماژول منبع باقی می‌ماند.

### ۳. میزکار باید Action-First باشد

داشبورد نباید صرفاً مجموعه‌ای از KPI و نمودار باشد. اولویت نمایش با موارد زیر است:

1. کارهای کاربر
2. تصمیم‌های نیازمند اقدام
3. جلسات و رویدادهای نزدیک
4. گفتگوهای خوانده‌نشده
5. گزارش فعالیت و حضور

### ۴. Hero میزکار باید کوتاه و ساختاریافته باشد

ساختار نهایی Hero:

- ردیف خوش‌آمدگویی و وضعیت حضور
- چهار کارت وضعیت هم‌اندازه
- جست‌وجوی سراسری تمام‌عرض

### ۵. Widgetها قابل شخصی‌سازی هستند

کاربر می‌تواند ترتیب Widgetها را تغییر دهد. در Prototype این وضعیت در Local Storage ذخیره می‌شود؛ در Production باید در تنظیمات کاربر Odoo ذخیره شود.

### ۶. ارتفاع Widgetها ثابت است

- Header ثابت
- Footer ثابت
- Body قابل Scroll
- فهرست‌های کم‌داده با ردیف خالی غیرفعال تکمیل می‌شوند

### ۷. گفتگو یک قابلیت سطح اول است

- عنوان نمایشی: «گفت‌وگوها»
- Route فنی برای سازگاری: `messages`
- دسترسی از Sidebar، Topbar و میزکار
- آیکن: دو حباب گفتگو
- اطلاعات اعضا، فایل‌ها و جزئیات در Drawer باز می‌شوند

### ۸. تقویم در میزکار تعاملی است

Widget تقویم باید بدون ترک میزکار بین روز، هفته و ماه تغییر کند. نمای ماهانه باید ماه، سال، روزهای هفته، روز جاری و روزهای دارای رویداد را مشخص کند.

### ۹. شخصی‌سازی ظاهری کنترل‌شده است

تنظیمات مجاز:

- اندازه متن
- تراکم رابط
- Accent Color
- Light/Dark Mode
- وضعیت Sidebar

شخصی‌سازی نباید ساختار دسترسی یا هویت سازمانی را بشکند.

### ۱۰. خوانایی بر تراکم اولویت دارد

هیچ متن کاربردی نباید کمتر از ۱۲ پیکسل باشد. حالت فشرده فقط فاصله‌ها را کاهش می‌دهد و نباید متن را ناخوانا کند.

### ۱۱. Sidebar باید واقعاً Collapse شود

در حالت جمع‌شده:

- کنترل بازکردن باید همیشه در دسترس باشد
- آیکن‌ها باقی بمانند
- Labelها مخفی شوند
- Tooltip برای مسیرها در نظر گرفته شود
- وضعیت کاربر حفظ شود

### ۱۲. دسترسی UI مرجع امنیت نیست

مخفی‌کردن منو و Route فقط UX است. مرجع نهایی امنیت:

- ACL
- Record Rule
- Method-level Checks
- Domain Scope

## موارد باز برای مرحله پیاده‌سازی

- مدل ذخیره تنظیمات شخصی Workspace
- قرارداد دقیق Widget Registry
- ساختار Server-side Drag Order
- اتصال Mail/Bus به گفتگوها
- اتصال Calendar واقعی
- Notification Service
- سیاست Theme سازمانی و محدودیت Accentها
- قرارداد Cache و Offline State
