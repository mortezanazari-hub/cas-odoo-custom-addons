# معماری و راهنمای استفاده از CAS Shift Management

## هدف و مرز

مدیریت سیاست حضور تاریخ‌دار، شیفت، الگوی چرخشی، انتساب کارکنان، تقویم روزانه و جابه‌جایی.

این ماژول مالک همین دامنه است و داده ماژول دیگر را فقط از رابطه/API مصرف می‌کند.

## وابستگی‌ها

- `hr`
- `mail`

## اجزای تشکیل‌دهنده

- `cas.attendance.policy`: سیاست محاسبه و تطبیق
- `cas.shift.template`: مشخصات یک نوبت
- `cas.shift.pattern` و line: چرخه کاری
- `cas.shift.assignment`: انتساب تاریخ‌دار
- `cas.shift.day`: snapshot روزانه
- `cas.official.holiday`: تعطیل رسمی
- `cas.shift.swap`: انتقال قانون دو روز

## نقش‌ها و امنیت

کاربر شیفت، برنامه‌ریز شیفت و مدیر شیفت/تقویم.

ACL، record rule شرکت/کارمند/حوزه و کنترل مدل، معیار نهایی‌اند؛ پنهان‌کردن منو امنیت ایجاد نمی‌کند.

## منو و رابط

ریشه شیفت، برنامه من، انتساب‌ها، جابه‌جایی‌ها و پیکربندی سیاست/الگو/تعطیلات.

Workspace می‌تواند مسیر ساده‌تری بدهد، اما محاسبه و مجوز در مدل‌های همین ماژول باقی می‌ماند.

## سناریوی عملیاتی

1. مدیر سیاست و شیفت پایه را تعریف می‌کند.
2. برنامه‌ریز الگو و خطوط چرخه را می‌سازد.
3. الگو برای بازه تاریخ‌دار به کارکنان/حوزه منتسب و منتشر می‌شود.
4. روزهای شیفت snapshot می‌شوند تا تغییر بعدی تاریخچه را عوض نکند.
5. در صورت نیاز قوانین دو روز با رکورد swap جابه‌جا و ممیزی می‌شوند.

## قواعد داده و ممیزی

- قواعد مؤثر باید تاریخ‌دار باشند.
- روز منتشرشده حذف یا بازنویسی تاریخی نمی‌شود.
- این ماژول تردد و کاردکس را محاسبه نمی‌کند.

اصلاح سوابق رسمی باید با ابطال، جایگزینی یا بازگشایی مجاز انجام شود؛ حذف فیزیکی سابقه عملیاتی مجاز نیست.

## نصب، ارتقا و تست

```bash
./odoo-bin -d <database> -i cas_shift_management --stop-after-init
./odoo-bin -d <database> -u cas_shift_management --stop-after-init
./odoo-bin -d <database> --test-enable --test-tags /cas_shift_management -u cas_shift_management --stop-after-init
```

آزمون‌های policy، pattern، assignment، snapshot و swap.

پس از ارتقا با کاربران نقش‌های مختلف، یک سناریوی کامل، جداسازی چندشرکتی، لاگ و فایل خروجی را کنترل کنید.

## راهنمای توسعه

- زمان محاسباتی را به دقیقه نگه دارید و فقط در UI/Excel قالب‌بندی کنید.
- قواعد مؤثر را تاریخ‌دار و سابقه رسمی را append-only طراحی کنید.
- عملیات import باید مرحله parse/review/commit و تشخیص تکراری داشته باشد.
- XML ID و نام مدل را بدون migration تغییر ندهید.
- هر تغییر محاسبه باید تست مرزی نیمه‌شب، تعطیل، قفل دوره و چندشرکتی داشته باشد.

## عیب‌یابی

- **نتیجه روز اشتباه است:** ابتدا shift day، سپس رخدادهای فعال و درخواست تأییدشده را بررسی کنید.
- **فایل وارد نمی‌شود:** قالب ستون‌ها، نگاشت هویت و ردیف‌های unmatched را ببینید.
- **رکورد قفل است:** وضعیت دوره و بازگشایی فعال را کنترل کنید.
- **کاربر داده‌ای نمی‌بیند:** employee-user، شرکت مجاز، نقش و record rule را بررسی کنید.
- **Excel ناسازگار است:** مجموع دقیقه‌ها را با سلول‌های duration مقایسه کنید، نه float ساعت.
