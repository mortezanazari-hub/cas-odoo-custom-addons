# معماری و راهنمای استفاده از CAS Workflow Core

## هدف و مرز

موتور نسخه‌دار گردش کار شامل حالت، گذار، نمونه اجرایی، مسئول، SLA و تاریخچه.

این ماژول مالک همین دامنه است؛ داده ماژول‌های دیگر باید با رابطه یا API مصرف شود و موازی بازسازی نشود.

## وابستگی‌ها

- `cas_form_core`
- `mail`

## اجزای تشکیل‌دهنده

- `cas.workflow.definition` و `cas.workflow.version`: تعریف و بازبینی
- `cas.workflow.state`: وضعیت
- `cas.workflow.transition`: گذار و قاعده
- `cas.workflow.instance`: اجرای متصل به منبع
- `cas.workflow.history`: تاریخچه append-only
- `cas.workflow.versioned.mixin`: قواعد نسخه

## نقش‌ها و امنیت

کاربر، طراح، منتشرکننده و مدیر گردش کار.

منو معیار نهایی امنیت نیست؛ ACL، record rule و کنترل سمت سرور باید دسترسی لینک مستقیم و RPC را نیز محدود کنند.

## رابط کاربری

نمونه‌ها، تعاریف، نسخه‌ها و پیکربندی گردش کار.

Workspace ممکن است مسیر ساده‌تری به همین قابلیت‌ها بدهد، اما مالکیت داده و مجوزها در این ماژول می‌ماند.

## سناریوی عملیاتی

1. تعریف و نسخه پیش‌نویس ساخته می‌شود.
2. حالت آغاز/میانی/پایان و گذارها تعریف می‌شوند.
3. نسخه پس از اعتبارسنجی منتشر می‌شود.
4. نمونه روی رکورد منبع شروع و به همان نسخه pin می‌شود.
5. مسئول جاری گذار مجاز را اجرا می‌کند؛ SLA و تاریخچه به‌روزرسانی می‌شوند.

## قواعد داده و ممیزی

- نمونه فعال به نسخه ثابت متصل است.
- فقط گذار مجاز برای کاربر و وضعیت جاری اجرا می‌شود.
- تاریخچه حذف یا بازنویسی نمی‌شود.

در عملیات رسمی از ابطال/بایگانی و رخداد جبرانی استفاده کنید؛ حذف فیزیکی نباید سابقه تصمیم را از بین ببرد.

## نصب، ارتقا و تست

```bash
./odoo-bin -d <database> -i cas_workflow_core --stop-after-init
./odoo-bin -d <database> -u cas_workflow_core --stop-after-init
./odoo-bin -d <database> --test-enable --test-tags /cas_workflow_core -u cas_workflow_core --stop-after-init
```

آزمون‌های نسخه، اجرا، دسترسی و تاریخچه.

پس از ارتقا، نقش‌ها، record rule، یک سناریوی کامل، لاگ سرور و assetهای دسکتاپ/موبایل را بررسی کنید.

## راهنمای توسعه

- منطق دامنه را در مدل/سرویس و UI را مصرف‌کننده نگه دارید.
- XML ID و نام مدل عمومی را بدون migration تغییر ندهید.
- عملیات رسمی را ممیزی‌پذیر و تست‌های batch/RPC را پوشش دهید.
- ادغام Action Hub فقط اقدام و لینک منبع را منتشر می‌کند و مالکیت رکورد را منتقل نمی‌کند.
- هر تغییر schema یا چرخه وضعیت باید همراه migration، آزمون و اصلاح این مستند باشد.

## عیب‌یابی

- **منو دیده نمی‌شود:** نقش، implied group و وابستگی‌ها را کنترل کنید.
- **رکورد باز نمی‌شود:** record rule شرکت، مالک و مسئول را بررسی کنید.
- **تغییر وضعیت رد می‌شود:** وضعیت جاری، نقش و فیلدهای الزامی را کنترل کنید.
- **UI قدیمی است:** ماژول را ارتقا، asset را بازسازی و کش مرورگر را پاک کنید.
- **نصب شکست خورد:** اولین traceback لاگ سرور را مبنا قرار دهید.

## مرجع فنی

فهرست دقیق مدل‌ها، فیلدها، متدها، ACL، record rule، منوها، actionها، cronها، assetها و آزمون‌ها در [مرجع فنی استخراج‌شده از کد](TECHNICAL_REFERENCE.md) نگهداری می‌شود.
