# معماری و راهنمای استفاده از CAS Correspondence

## هدف و مرز

مکاتبات رسمی داخلی، گیرنده، ارجاع، مشاهده، پاسخ/اصلاح، محرمانگی و ممیزی؛ جدا از چت.

این ماژول مالک همین دامنه است؛ داده ماژول‌های دیگر باید با رابطه یا API مصرف شود و موازی بازسازی نشود.

## وابستگی‌ها

- `cas_core`
- `mail`
- `hr`

## اجزای تشکیل‌دهنده

- `cas.correspondence.letter`: نامه
- `cas.correspondence.recipient`: گیرنده
- `cas.correspondence.referral`: ارجاع
- `cas.correspondence.view.receipt`: رسید مشاهده
- `cas.correspondence.relation`: پاسخ/اصلاح
- `cas.correspondence.audit`: ممیزی
- `cas.correspondence.secretariat.delegation`: نمایندگی
- wizardهای ارجاع، تکمیل و لغو

## نقش‌ها و امنیت

کاربر مکاتبات و مدیر دبیرخانه.

منو معیار نهایی امنیت نیست؛ ACL، record rule و کنترل سمت سرور باید دسترسی لینک مستقیم و RPC را نیز محدود کنند.

## رابط کاربری

نامه‌ها، پیش‌نویس‌های من، دبیرخانه و نمایندگی‌ها.

Workspace ممکن است مسیر ساده‌تری به همین قابلیت‌ها بدهد، اما مالکیت داده و مجوزها در این ماژول می‌ماند.

## سناریوی عملیاتی

1. نامه با محرمانگی و گیرندگان آماده می‌شود.
2. ارسال، دسترسی گیرندگان و سابقه رسمی را تثبیت می‌کند.
3. گیرنده مشاهده را ثبت یا نامه را ارجاع می‌دهد.
4. مسئول ارجاع کار را شروع و تکمیل می‌کند.
5. پاسخ/اصلاح با رابطه به اصل ساخته و سپس نامه بسته می‌شود.

## قواعد داده و ممیزی

- record rule و محرمانگی مرجع دسترسی‌اند.
- ارسال، مشاهده، ارجاع و اصلاح ممیزی می‌شوند.
- چت جای نامه رسمی نیست.

در عملیات رسمی از ابطال/بایگانی و رخداد جبرانی استفاده کنید؛ حذف فیزیکی نباید سابقه تصمیم را از بین ببرد.

## نصب، ارتقا و تست

```bash
./odoo-bin -d <database> -i cas_correspondence --stop-after-init
./odoo-bin -d <database> -u cas_correspondence --stop-after-init
./odoo-bin -d <database> --test-enable --test-tags /cas_correspondence -u cas_correspondence --stop-after-init
```

آزمون‌های security، lifecycle، referral، delegation و audit.

پس از ارتقا، نقش‌ها، record rule، یک سناریوی کامل، لاگ سرور و assetهای دسکتاپ/موبایل را بررسی کنید.

## راهنمای توسعه

- منطق دامنه را در مدل/سرویس و UI را مصرف‌کننده نگه دارید.
- XML ID و نام مدل عمومی را بدون migration تغییر ندهید.
- عملیات رسمی را ممیزی‌پذیر و تست‌های batch/RPC را پوشش دهید.
- ادغام Action Hub فقط اقدام و لینک منبع را منتشر می‌کند و مالکیت رکورد را منتقل نمی‌کند.
- هر تغییر schema یا چرخه وضعیت باید همراه migration، آزمون و اصلاح این مستند باشد.

## عیب‌یابی

- **منو دیده نمی‌شود:** نقش، implied group و وابستگی‌ها را کنترل کنید.
- **رکورد باز نمی‌شود:** record rule شرکت، مالک و مسئول را بررسی کنید.
- **تغییر وضعیت رد می‌شود:** وضعیت جاری، نقش و فیلدهای الزامی را کنترل کنید.
- **UI قدیمی است:** ماژول را ارتقا، asset را بازسازی و کش مرورگر را پاک کنید.
- **نصب شکست خورد:** اولین traceback لاگ سرور را مبنا قرار دهید.
