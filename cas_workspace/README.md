# CAS Organizational Workspace

> نسخه: `19.0.2.0.1` · Odoo 19 Community

پوسته اختصاصی RTL برای دسترسی یکپارچه به قابلیت‌های CAS، داشبورد کارتابل، صفحات داده و ناوبری ثابت سمت راست.

## اجزای ماژول

- `cas.workspace.dashboard`: سرویس داشبورد، ناوبری، صفحات و جزئیات
- `PAGE_CONFIGS`: نگاشت routeها به مدل، ستون، domain و ترتیب
- `workspace.js/xml`: client action، router داخلی، جدول و drawer
- `workspace.scss`: sidebar، scroll و responsive
- `login_theme.scss`: ظاهر صفحه ورود
- action و منوی ریشه «فضای کار سازمانی»

## نقش‌ها و امنیت

Workspace نقش کسب‌وکاری تازه نمی‌سازد و باید از ACL و record rule ماژول‌های منبع تبعیت کند. نمایش یک route یا دکمه مجوز دسترسی به رکورد ایجاد نمی‌کند.

## روش کار

1. کاربر «فضای کار سازمانی» را باز می‌کند.
2. داشبورد آمار Action Hub و آخرین مکاتبات مجاز را نشان می‌دهد.
3. منوی سمت راست route داخلی را عوض و صفحه اختصاصی همان حوزه را بارگذاری می‌کند.
4. جستجو، صفحه‌بندی و drawer جزئیات از سرویس backend استفاده می‌کنند.
5. sidebar در دسکتاپ collapse/expand و در موبایل off-canvas است.

## دامنه ناوبری

کارتابل، اقدام فوری، فرم‌ها، گردش‌کار، مکاتبات، اسناد، تأییدها، حضور، عملیات تردد، شیفت، کاردکس، گزارش کار و تنظیمات.

## محدودیت‌های فعلی

- صفحه Settings فعلی نمای کاربران داخلی و ماژول‌های نصب‌شده است؛ جایگزین کامل `res.config.settings` نیست.
- فرم‌ساز و طراح گردش‌کار ابزار طراحی‌اند. در UX نهایی بهتر است از بستر «تعریف فرایند» و فقط برای نقش طراح باز شوند، نه به‌عنوان مقصد روزمره همه کاربران.
- هر route فعلاً یک صفحه داده عمومی می‌سازد؛ عملیات تخصصی هر دامنه باید مرحله‌به‌مرحله با UX همان ماژول تکمیل شود.

## نصب و ارتقا

```bash
./odoo-bin -d <database> -i cas_workspace --stop-after-init
./odoo-bin -d <database> -u cas_workspace --stop-after-init
```

برای جزئیات معماری، routeها، امنیت و QA، [راهنمای کامل](docs/ARCHITECTURE_AND_USAGE.md) را ببینید.
