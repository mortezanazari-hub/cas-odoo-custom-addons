# معماری و راهنمای استفاده از CAS Daily Work Reports

## هدف و مرز

گزارش روزانه کار کارکنان با ایستگاه، نمایندگی، مهلت، تأیید رسمی و Excel.

این ماژول مالک همین دامنه است و داده ماژول دیگر را فقط از رابطه/API مصرف می‌کند.

## وابستگی‌ها

- `hr`
- `mail`
- `cas_workflow_core`
- `cas_approval_core`

## اجزای تشکیل‌دهنده

- `cas.work.station`: ایستگاه کاری مستقل از واحد سازمانی
- `cas.work.report`: گزارش روزانه و ساعات
- `cas.work.report.delegation`: مجوز نمایندگی تاریخ‌دار
- `cas.work.report.export.wizard`: Excel
- افزونه `hr.employee`
- workflow/approval پیش‌فرض در post-init

## نقش‌ها و امنیت

کاربر، سرپرست و مدیر گزارش کار.

ACL، record rule شرکت/کارمند/حوزه و کنترل مدل، معیار نهایی‌اند؛ پنهان‌کردن منو امنیت ایجاد نمی‌کند.

## منو و رابط

گزارش‌های من، گزارش‌های حوزه من، خروجی Excel، ایستگاه‌ها و نمایندگی‌ها.

Workspace می‌تواند مسیر ساده‌تری بدهد، اما محاسبه و مجوز در مدل‌های همین ماژول باقی می‌ماند.

## سناریوی عملیاتی

1. کارمند یا نماینده مجاز گزارش روز/شیفت را در ایستگاه ثبت می‌کند.
2. ساعت شروع/پایان، کار عادی، اضافه‌کاری و شرح فعالیت تکمیل می‌شود.
3. گزارش حداکثر تا مهلت ۱۲ ساعته ارسال می‌شود.
4. تأیید رسمی سرپرست از Approval/Workflow انجام می‌شود؛ ثبت توسط مدیر مستقیم می‌تواند خودکار تأیید شود.
5. کاربر مجاز خروجی Excel حوزه قابل مشاهده را می‌گیرد.

## قواعد داده و ممیزی

- نمایندگی باید صریح، تاریخ‌دار و متناسب با حوزه باشد.
- دیدن گزارش تابع زنجیره سازمانی و record rule است.
- شماره، chatter و تاریخچه تأیید حفظ می‌شوند.

اصلاح سوابق رسمی باید با ابطال، جایگزینی یا بازگشایی مجاز انجام شود؛ حذف فیزیکی سابقه عملیاتی مجاز نیست.

## نصب، ارتقا و تست

```bash
./odoo-bin -d <database> -i cas_work_report --stop-after-init
./odoo-bin -d <database> -u cas_work_report --stop-after-init
./odoo-bin -d <database> --test-enable --test-tags /cas_work_report -u cas_work_report --stop-after-init
```

آزمون‌های deadline، visibility، delegation، approval و export.

پس از ارتقا با کاربران نقش‌های مختلف، یک سناریوی کامل، جداسازی چندشرکتی، لاگ و فایل خروجی را کنترل کنید.

## راهنمای توسعه

- زمان محاسباتی را به دقیقه نگه دارید و فقط در UI/Excel قالب‌بندی کنید.
- قواعد مؤثر را تاریخ‌دار و سابقه رسمی را append-only طراحی کنید.
- عملیات import باید مرحله parse/review/commit و تشخیص تکراری داشته باشد.
- XML ID و نام مدل را بدون migration تغییر ندهید.
- هر تغییر محاسبه باید تست مرزی نیمه‌شب، تعطیل، قفل دوره و چندشرکتی داشته باشد.

## عیب‌یابی

- **نتیجه روز اشتباه است:** ابتدا shift day، سپس رخدادهای فعال و درخواست تأییدشده را بررسی کنید.
- **فایل وارد نمی‌شود:** قالب ستون‌ها، نگاشت هویت و ردیف‌های unmatched را ببینید.
- **رکورد قفل است:** وضعیت دوره و بازگشایی فعال را کنترل کنید.
- **کاربر داده‌ای نمی‌بیند:** employee-user، شرکت مجاز، نقش و record rule را بررسی کنید.
- **Excel ناسازگار است:** مجموع دقیقه‌ها را با سلول‌های duration مقایسه کنید، نه float ساعت.
