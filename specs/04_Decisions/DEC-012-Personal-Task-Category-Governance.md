# DEC-012 — حاکمیت دسته‌های کار شخصی

| مشخصه | مقدار |
|---|---|
| وضعیت | `Agreed` |
| نسخه | `CAS UI Workspace v8` |
| دامنه | Personal Task / Category / Ownership |
| صفحات مرتبط | `02_UI_UX/Employee/Personal_Tasks.md` |

## زمینه

صفحه «کارهای من» در نسخه ۷ مفهوم Task شخصی را از Action سازمانی جدا کرد، اما مدیریت دسته‌ها مشخص نبود. نسخه ۸ باید میان دسته‌های سیستمی و دسته‌های شخصی تمایز قطعی ایجاد کند.

## تصمیم

1. دسته‌های سیستمی توسط سامانه ایجاد می‌شوند و کاربر عادی حق حذف یا تغییر هویت آن‌ها را ندارد.
2. دسته‌های شخصی متعلق به کاربر هستند و ایجاد، تغییر نام، تغییر ترتیب و حذف آن‌ها مجاز است.
3. حذف دسته شخصی نباید موجب حذف Taskهای آن شود. Taskها به دسته پیش‌فرض شخصی منتقل می‌شوند.
4. نام دسته در Scope همان کاربر یکتا است؛ مقایسه باید پس از Trim و Normalization انجام شود.
5. دسته‌های شخصی دیگر کاربران به‌صورت پیش‌فرض قابل مشاهده یا مدیریت نیستند.
6. دسته در نسخه نخست صرفاً ابزار سازمان‌دهی است و مجوز، Workflow یا Approval تولید نمی‌کند.

## دسته‌های سیستمی حداقلی

- همه
- امروز
- برنامه‌ریزی‌شده
- تکمیل‌شده
- شخصی یا بدون دسته

نام نمایشی قابل ترجمه است، اما شناسه فنی دسته سیستمی پایدار می‌ماند.

## قواعد حذف

- حذف دسته سیستمی ممنوع است.
- حذف دسته شخصی نیازمند تأیید کاربر است.
- تعداد Taskهای متأثر پیش از تأیید نمایش داده می‌شود.
- انتقال Taskها و حذف دسته باید در یک Transaction انجام شود.
- حذف دسته نباید Task، Reminder یا پیوند گزارش کار را حذف کند.

## امنیت

کنترل مالکیت و ممنوعیت ویرایش دسته سیستمی باید در مدل و Method سمت سرور اعمال شود. مخفی‌کردن دکمه در UI کافی نیست.

## پیامد ماژولی

- `cas_personal_task` در صورت ایجاد، مالک Category و Task خواهد بود.
- در صورت نگهداری موقت داخل `cas_workspace`، مدل‌ها باید به‌گونه‌ای طراحی شوند که بعداً بدون شکستن شناسه‌ها قابل استخراج باشند.
- `cas_action_hub` نباید دسته‌های Task شخصی را مصرف یا مدیریت کند.

## معیار پذیرش

- دسته شخصی قابل افزودن، تغییر نام و حذف باشد.
- دسته سیستمی فاقد عملیات حذف و تغییر نام باشد.
- با حذف دسته شخصی، Taskها سالم و قابل دسترسی باقی بمانند.
- درخواست مستقیم RPC برای حذف دسته سیستمی رد شود.
