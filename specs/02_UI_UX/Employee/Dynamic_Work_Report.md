# صفحه پویا گزارش روزانه

| مشخصه | مقدار |
|---|---|
| شناسه | `PAGE-EMP-WR-001` |
| نسخه هدف | Workspace v8 Iteration 12 |
| وضعیت محصول | `Agreed` |
| وضعیت معماری | `Needs Review` |
| ماژول مالک دامنه | `cas_work_report` |

## هدف
نمایش فرم مناسب گزارش کار براساس Profile مؤثر کاربر، بدون ساخت صفحه و ماژول مستقل برای هر فرم.

## ساختار ثابت صفحه
1. عنوان فرم و کد Profile
2. توضیح دلیل انتخاب فرم
3. اطلاعات خودکار کارمند، واحد، شغل، شیفت و حضور
4. Runtime فرم تخصصی
5. موانع، پیگیری و شواهد
6. وضعیت تکمیل، مهلت و مسیر تأیید
7. نسخه فرم و Snapshot

## بخش‌های پویا
- عنوان و راهنمای فرم
- Label و نوع فیلدها
- Sectionها و ترتیب آن‌ها
- جدول یا کارت فعالیت‌ها
- Validation و منطق شرطی
- Evidence Policy
- Reviewer و Deadline نمایش‌داده‌شده
- Renderer اختصاصی در صورت تعریف

## قواعد نقش‌ها
### کاربر عادی
- اطلاعات هویتی Readonly
- ثبت فقط برای خود، مگر Delegation صریح
- ذخیره پیش‌نویس و ارسال

### سرپرست
- صفحه ثبت گزارش شخصی مستقل از نمای بررسی تیم
- دسترسی به Return/Approve فقط در Review Route

### مدیر و مدیرعامل
- فرم شخصی در صورت الزام Profile
- نمای اصلی مدیریتی، تجمیعی است و فرم خام فقط با Drill-down باز می‌شود

### نگهبان و نقش‌های تخصصی
- Sectionها و واژگان متناسب با شغل
- نمونه نگهبانی: رخداد، گشت، تردد، تماس و تحویل شیفت
- نمونه IT: تیکت، سامانه، اقدام، زمان رفع و نتیجه

## رفتار Profile Banner
Banner بالای فرم باید نشان دهد:
- نام فرم فعال
- کد Profile
- واحد/شغل یا Assignment مؤثر
- نسخه فرم
- تاریخ اعتبار
- وضعیت «پویا»

## دلیل نمایش فرم
کاربر باید بتواند بفهمد چرا این فرم انتخاب شده است. UI توضیح کوتاه می‌دهد و جزئیات فنی فقط برای مدیر سامانه قابل مشاهده است.

## خطاهای کنترل‌شده
- Profile پیدا نشد
- چند Profile هم‌اولویت
- Form Version منتشرشده وجود ندارد
- Assignment معتبر نیست
- Provider شیفت یا حضور موقتاً در دسترس نیست

در این حالت‌ها فرم اشتباه یا پیش‌فرض تصادفی نمایش داده نمی‌شود.

## معیارهای پذیرش
- تغییر نقش Prototype ساختار تخصصی فرم را تغییر دهد.
- نام کارمند و Context سازمانی به‌صورت فقط‌خواندنی باشند.
- نسخه فرم و Profile قابل مشاهده باشند.
- صفحه از آخرین Form Version منتشرشده برای گزارش جدید استفاده کند.
- گزارش موجود همیشه با Snapshot و Version خودش باز شود.
- Payload دستکاری‌شده نتواند Profile یا Employee دیگری را تحمیل کند.
- نمای موبایل قابل استفاده و جدول‌های عریض دارای Scroll داخلی باشند.
