# معماری و راهنمای استفاده از CAS Approval Core

## هدف و مرز

سیاست تأیید چندمرحله‌ای، تعیین تأییدکننده، نمایندگی، تصمیم و تاریخچه رسمی.

این ماژول مالک همین دامنه است؛ داده ماژول‌های دیگر باید با رابطه یا API مصرف شود و موازی بازسازی نشود.

## وابستگی‌ها

- `cas_workflow_core`
- `mail`
- `hr`

## اجزای تشکیل‌دهنده

- `cas.approval.policy` و `cas.approval.step`: سیاست و مرحله
- `cas.approval.request` و `cas.approval.line`: درخواست و تصمیم
- `cas.approval.history`: دفتر ممیزی
- `cas.approval.delegation`: جانشینی تاریخ‌دار
- wizard رد
- افزونه workflow version/instance

## نقش‌ها و امنیت

کاربر تأیید و مدیر تأیید؛ مدیر سیستم نقش مدیر دارد.

منو معیار نهایی امنیت نیست؛ ACL، record rule و کنترل سمت سرور باید دسترسی لینک مستقیم و RPC را نیز محدود کنند.

## رابط کاربری

صندوق تأیید، درخواست‌ها، سیاست‌ها و نمایندگی‌ها.

Workspace ممکن است مسیر ساده‌تری به همین قابلیت‌ها بدهد، اما مالکیت داده و مجوزها در این ماژول می‌ماند.

## سناریوی عملیاتی

1. مدیر سیاست و مراحل را تعریف می‌کند.
2. درخواست ایجاد و تأییدکننده هر مرحله resolve می‌شود.
3. تأییدکننده یا نماینده معتبر تصمیم می‌گیرد.
4. تأیید مرحله بعد/گذار را فعال می‌کند؛ رد دلیل می‌خواهد.
5. همه تصمیم‌ها در تاریخچه می‌مانند.

## قواعد داده و ممیزی

- تصمیم فقط توسط مسئول یا نماینده معتبر است.
- رد بدون دلیل ثبت نمی‌شود.
- سوابق رسمی بازنویسی نمی‌شوند.

در عملیات رسمی از ابطال/بایگانی و رخداد جبرانی استفاده کنید؛ حذف فیزیکی نباید سابقه تصمیم را از بین ببرد.

## نصب، ارتقا و تست

```bash
./odoo-bin -d <database> -i cas_approval_core --stop-after-init
./odoo-bin -d <database> -u cas_approval_core --stop-after-init
./odoo-bin -d <database> --test-enable --test-tags /cas_approval_core -u cas_approval_core --stop-after-init
```

آزمون‌های policy، delegation، approve/reject و workflow integration.

پس از ارتقا، نقش‌ها، record rule، یک سناریوی کامل، لاگ سرور و assetهای دسکتاپ/موبایل را بررسی کنید.

## راهنمای توسعه

- منطق دامنه را در مدل/سرویس و UI را مصرف‌کننده نگه دارید.
- XML ID و نام مدل عمومی را بدون migration تغییر ندهید.
- عملیات رسمی را ممیزی‌پذیر و تست‌های batch/RPC را پوشش دهید.
- ادغام Action Hub فقط اقدام و لینک منبع را منتشر می‌کند و مالکیت رکورد را منتقل نمی‌کند.
- هر تغییر schema یا چرخه وضعیت باید همراه migration، آزمون و اصلاح این مستند باشد.

## عیب‌یابی

- **منو دیده نمی‌شود:** نقش، implied group و وابستگی‌ها را کنترل کنید.
- **رکورد باز نمی‌شود:** record rule شرکت، مالک و مسئول را بررسی کنید.
- **تغییر وضعیت رد می‌شود:** وضعیت جاری، نقش و فیلدهای الزامی را کنترل کنید.
- **UI قدیمی است:** ماژول را ارتقا، asset را بازسازی و کش مرورگر را پاک کنید.
- **نصب شکست خورد:** اولین traceback لاگ سرور را مبنا قرار دهید.
