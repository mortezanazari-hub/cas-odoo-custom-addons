# معماری و راهنمای استفاده از CAS Dynamic Form

## هدف و مرز

اجرای دیداری و RTL فرم‌های منتشرشده با OWL، ذخیره پیش‌نویس و ورودی جلالی.

این ماژول مالک همین دامنه است؛ داده ماژول‌های دیگر باید با رابطه یا API مصرف شود و موازی بازسازی نشود.

## وابستگی‌ها

- `cas_form_core`
- `cas_jalali`
- `web`

## اجزای تشکیل‌دهنده

- افزونه مدل‌های تعریف، نسخه، فیلد و submission
- `dynamic_form_app.js/xml`: برنامه اجرا
- `dynamic_reference_field.js/xml`: انتخاب رکورد مرجع
- `dynamic_form.scss`: ظاهر واکنش‌گرا
- Client action با تگ `action_cas_dynamic_form_runtime`

## نقش‌ها و امنیت

کاربر مجاز ثبت فرم؛ انتشار و طراحی تابع Form Core.

منو معیار نهایی امنیت نیست؛ ACL، record rule و کنترل سمت سرور باید دسترسی لینک مستقیم و RPC را نیز محدود کنند.

## رابط کاربری

از فرم منتشرشده یا Workspace باز می‌شود؛ منوی مدیریتی مستقل لازم ندارد.

Workspace ممکن است مسیر ساده‌تری به همین قابلیت‌ها بدهد، اما مالکیت داده و مجوزها در این ماژول می‌ماند.

## سناریوی عملیاتی

1. کاربر یک نسخه منتشرشده را باز می‌کند.
2. Runtime schema را بارگذاری و کنترل مناسب هر نوع فیلد را نمایش می‌دهد.
3. پاسخ‌ها پیش‌نویس ذخیره و بازیابی می‌شوند.
4. ارسال نهایی اعتبارسنجی سمت کاربر و سرور را اجرا می‌کند.
5. تاریخ برای کاربر جلالی و در Odoo به‌صورت استاندارد میلادی/UTC ذخیره می‌شود.

## قواعد داده و ممیزی

- Runtime فقط نسخه منتشرشده و مجاز را اجرا می‌کند.
- اعتبارسنجی سرور مرجع نهایی است.
- نمایش جلالی مقدار ذخیره‌شده را تغییر نمی‌دهد.

در عملیات رسمی از ابطال/بایگانی و رخداد جبرانی استفاده کنید؛ حذف فیزیکی نباید سابقه تصمیم را از بین ببرد.

## نصب، ارتقا و تست

```bash
./odoo-bin -d <database> -i cas_dynamic_form --stop-after-init
./odoo-bin -d <database> -u cas_dynamic_form --stop-after-init
./odoo-bin -d <database> --test-enable --test-tags /cas_dynamic_form -u cas_dynamic_form --stop-after-init
```

آزمون‌های runtime در `tests/`.

پس از ارتقا، نقش‌ها، record rule، یک سناریوی کامل، لاگ سرور و assetهای دسکتاپ/موبایل را بررسی کنید.

## راهنمای توسعه

- منطق دامنه را در مدل/سرویس و UI را مصرف‌کننده نگه دارید.
- XML ID و نام مدل عمومی را بدون migration تغییر ندهید.
- عملیات رسمی را ممیزی‌پذیر و تست‌های batch/RPC را پوشش دهید.
- ادغام Action Hub فقط اقدام و لینک منبع را منتشر می‌کند و مالکیت رکورد را منتقل نمی‌کند.
- هر تغییر schema یا چرخه وضعیت باید همراه migration، آزمون و اصلاح این مستند باشد.

## عیب‌یابی

- **منو دیده نمی‌شود:** نقش، implied group و وابستگی‌ها را کنترل کنید.
- **رکورد باز نمی‌شود:** record rule شرکت، مالک و مسئول را بررسی کنید.
- **تغییر وضعیت رد می‌شود:** وضعیت جاری، نقش و فیلدهای الزامی را کنترل کنید.
- **UI قدیمی است:** ماژول را ارتقا، asset را بازسازی و کش مرورگر را پاک کنید.
- **نصب شکست خورد:** اولین traceback لاگ سرور را مبنا قرار دهید.
